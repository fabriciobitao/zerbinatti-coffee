# Auditoria de Acessibilidade — Zerbinatti Coffee

Data: 2026-05-03
Auditor: AccessibilityAuditor (Claude Opus 4.7)
Escopo: Home (`/`) + páginas estáticas (`/termos`, `/privacidade`, `/entregas`, `/fazenda`, `/processo`)
Padrão: **WCAG 2.1 AA** (obrigatório no Brasil — Lei 13.146/2015 LBI, art. 63)

## 3. Issues críticos (bloqueia acesso — corrigir antes do release)

### CRIT-01 · Eyebrow olive sobre fundo ink reprova WCAG 1.4.3
**WCAG**: 1.4.3 Contrast Minimum (AA)
**Componente**: `Subscription.tsx:112` — `<p className="eyebrow">O CARTÃO DA CASA</p>`
**Problema**: `.eyebrow` tem `color: var(--olive)` definido em `globals.css:151`. Sobre `bg-ink`, ratio = **2.11:1** — falha em qualquer tamanho.
**Fix**: criar variante `.eyebrow-on-dark { color: var(--bone); }` ou usar `--bone-soft` (ratio 14:1). A UI-SPEC seção 3.4 já antecipou essa decisão; falta aplicar.
```tsx
<p className="eyebrow text-bone-soft">O CARTÃO DA CASA</p>
```

### CRIT-02 · Sub-preço e caption legal ilegíveis no configurador
**WCAG**: 1.4.3 Contrast Minimum (AA)
**Componente**: `Subscription.tsx:230, 254, 269` — `text-ink-mute` sobre `bg-ink-soft` (radio inativo, sub-total mono, legal-caption)
**Problema**: ratio **2.05:1**. Texto legal ("Cobrança recorrente...") é **obrigatório por CDC** estar legível.
**Fix**: trocar para `text-bone-soft` ou novo token `--ink-mute-on-dark` (ex: `#A39C92` → ratio 5.0:1).

### CRIT-03 · Páginas estáticas (StaticPage) usam paleta antiga reprovada
**WCAG**: 1.4.3 Contrast Minimum (AA)
**Componente**: `StaticPage.tsx:18,28,42` — `bg-coffee-50`, `text-gold-600`, `text-coffee-700`, `text-coffee-600`
**Problema**: o `globals.css` mapeou os tokens legados para a nova paleta (linhas 70-89). Resultado: `.text-coffee-600` agora resolve para `--ink-mute (#736961)`. Sobre `bg-coffee-50` (= `--bone`), ratio = **4.41:1** — falha AA Normal por 0.09. Eyebrow `text-gold-600` virou `--olive` — passa. Headers H2 usam `text-coffee-900` = `--ink` — passa. **Mas o body inteiro usa `text-coffee-700` = `--ink-soft` que passa.** O risco real está no link "Voltar para a loja" (`text-coffee-600`) e a data "Última atualização" — falham AA Normal.
**Fix**: trocar `text-coffee-600`/`text-coffee-500` para `text-ink-soft` no StaticPage. Páginas afetadas: `/termos`, `/privacidade`, `/entregas`, `/fazenda`, `/processo`.

### CRIT-04 · CartDrawer inteiro fora da paleta nova + contrastes ruins
**WCAG**: 1.4.3 (Contrast), 4.1.2 (Name/Role/Value), 2.4.3 (Focus Order)
**Componente**: `CartDrawer.tsx` (todo o arquivo)
**Problemas múltiplos**:
- Usa `bg-coffee-800`, `bg-gold-500`, `bg-green-800`, `text-red-600` (cores que **não existem** na paleta canônica — só vivem como aliases legados)
- Cor única para indicar formato do SKU (`getAccentColor`) — falha **1.4.1 Use of Color**
- Diálogo modal com `role="dialog"` mas **sem focus trap** — `Tab` escapa para o body atrás
- **Sem `aria-labelledby`** apontando para o `<h3>` "Seu carrinho"
- Botão de fechar tem só ícone SVG, **sem `aria-label`** (`<button onClick=...><svg/></button>`)
- Botão `−`/`+` de quantidade sem `aria-label` (`Diminuir quantidade`/`Aumentar quantidade`)
- Botão remover item sem `aria-label`
- Anúncio "Frete grátis aplicado" não está em `aria-live`
- Cores `text-coffee-400` (mapeado a `#A39C92`) sobre `bg-white` ratio = **2.5:1** — FALHA
**Fix**: reescrever CartDrawer no sistema bone/ink/olive + adicionar focus trap (`focus-trap-react` ou implementação manual com `useEffect` + `tabindex`), aria-labels em todos os botões-ícone, aria-labelledby no dialog.

### CRIT-05 · Sem skip link "Pular para o conteúdo"
**WCAG**: 2.4.1 Bypass Blocks (A — não AA, mas obrigatório nível A)
**Componente**: `layout.tsx`
**Problema**: usuários de teclado e leitor de tela precisam tabular por todo o Header (logo + 4 nav + cart + menu) em cada página.
**Fix**:
```tsx
<body>
  <a href="#main" className="sr-only focus:not-sr-only focus:fixed focus:top-4 focus:left-4 focus:z-[100] focus:bg-ink focus:px-4 focus:py-2 focus:text-bone">
    Pular para o conteúdo
  </a>
  ...
</body>
```
E adicionar `id="main"` no `<main>` da página.

### CRIT-06 · Mobile menu sem focus trap nem fechamento por Esc
**WCAG**: 2.1.2 No Keyboard Trap (inverso — sem trap correto), 2.4.3 Focus Order
**Componente**: `Header.tsx:119-140`
**Problema**: ao abrir o menu mobile, foco continua no body atrás. Tab leva o usuário para conteúdo invisível abaixo do menu. **Esc não fecha.** O menu também não retorna foco ao botão hamburguer ao fechar.
**Fix**: adicionar `useEffect` com listener de Escape, mover foco para o primeiro link ao abrir, retornar ao botão ao fechar, e `inert` no resto da página enquanto aberto.

## 4. Issues altos (barreiras significativas)

### HIGH-01 · `role="list"` + `role="listitem"` redundantes em Cafes.tsx
**WCAG**: 4.1.1 Parsing / boas práticas ARIA
**Componente**: `Cafes.tsx:123-129`
**Problema**: `<div role="list">` com filhos `<article role="listitem">`. **Articles dentro de div com role=list** é anti-pattern: alguns leitores de tela (VoiceOver) anunciam tanto "lista" quanto "artigo", criando ruído. Além disso, `<article>` perde sua semântica natural quando vira `listitem`.
**Fix**: usar `<ul>` + `<li>` envolvendo os `<article>`, **ou** remover os `role`s e deixar `<article>` puros com landmark navigation natural.

### HIGH-02 · Toggle de frequência e radio cards não respondem a Setas (padrão WAI-ARIA)
**WCAG**: 2.1.1 Keyboard, 4.1.2 Name Role Value
**Componente**: `Subscription.tsx:165-238`
**Problema**: os radios estão dentro de `<label>` com `<input type="radio" className="sr-only">`. Tecnicamente o teclado funciona (Tab + Espaço), **mas o padrão correto para radio group é Setas mudarem seleção sem Tab entre opções**. Como hoje cada label é tabbable individualmente, o usuário tabula 5 vezes só para configurar (2 freq + 3 pacote). O esperado: 1 Tab para cada fieldset, Setas dentro.
**Fix**: usar `tabindex={active ? 0 : -1}` nos labels e listener de KeyDown nas Setas (Up/Down/Left/Right) para mudar seleção. Ou aceitar o trade-off documentado e adicionar instrução visual ("Use Tab e Espaço").

### HIGH-03 · NewsletterForm sem mensagem de erro acessível
**WCAG**: 3.3.1 Error Identification, 3.3.3 Error Suggestion
**Componente**: `NewsletterForm.tsx`
**Problema**: o `try/catch` engole o erro silenciosamente e marca como `sent` mesmo se a API falhar (linha 21 comment: "Fail silent"). **Isso é mentira para o usuário** — ele acha que assinou e nunca receberá nada. Sem `aria-invalid`, sem `aria-describedby` apontando para erro, sem `role="alert"`.
**Fix**: estado de erro real, `<p role="alert" aria-live="assertive">` para mensagem, `aria-invalid` no input, e remover o "fail silent".

### HIGH-04 · Total da assinatura não é anunciado de forma confiável
**WCAG**: 4.1.3 Status Messages
**Componente**: `Subscription.tsx:244` — `<div aria-live="polite">`
**Problema**: o `aria-live` envolve **toda a área** do total. Quando o usuário muda só a frequência, o número muda mas o leitor pode ler tudo de novo (label + valor + sub-label). Pior: alguns leitores não disparam re-anúncio se só os filhos mudam mas o container já existia.
**Fix**: criar um `<span aria-live="polite" aria-atomic="true">` apenas no valor formatado, e um `<span className="sr-only">` que mude o texto descritivo ("Total atualizado: R$ 76,42 por envio mensal de 500g em grãos") — força reanúncio limpo.

### HIGH-05 · Hero H1 com `<br>` semântico questionável
**WCAG**: 1.3.1 Info and Relationships
**Componente**: `Hero.tsx:24-29`
**Problema**: `<h1>Café especial brasileiro,<br /><em>casa italiana.</em></h1>`. O `<br>` é decorativo (controlar quebra), mas em leitores de tela mobile pode quebrar a frase em duas. Não é fail crítico, mas leitor anuncia pausa indevida.
**Fix**: remover `<br>` e controlar quebra via CSS `display: block` no `<em>` ou `::after`. Melhor: usar `<span className="block">` em vez de `<br>`.

### HIGH-06 · Imagens decorativas Unsplash com alt mas tratadas como conteúdo
**WCAG**: 1.1.1 Non-text Content
**Componente**: `Cafes.tsx:36,52,69` + `Story1897.tsx:19`
**Problema**: as URLs apontam para `images.unsplash.com` — a UI-SPEC explicitamente **veta foto stock** (seção 1.4 Hero, seção 4.4 História). O alt está ok, mas se as 3 fotos do Cafes são literalmente fotos genéricas de café qualquer (não os pacotes Zerbinatti), o alt é **misleading** ("Pacote 500g de Café Zerbinatti em grãos sobre mesa de madeira" descreve um pacote que não está na foto).
**Fix**: substituir por fotos reais do produto OU usar alt genérico ("Café em grãos sobre mesa de madeira") sem mentir sobre o conteúdo. Brand + a11y andam juntos aqui.

### HIGH-07 · Logo wordmark "Zerbinatti" e logo image têm alt redundante
**WCAG**: 1.1.1 Non-text Content
**Componente**: `Header.tsx:50-66`
**Problema**: `<Link aria-label="Zerbinatti Coffee, ir para a home">` envolve um `<img alt="Zerbinatti Coffee">` ou wordmark. O leitor anuncia "Zerbinatti Coffee, ir para a home, link, imagem, Zerbinatti Coffee" — duplicado.
**Fix**: `<img alt="">` (decorativo) já que o link tem aria-label, OU remover aria-label e deixar o alt da imagem fazer o trabalho.

### HIGH-08 · Subscription quiz link href="#" é placeholder quebrado
**WCAG**: 2.4.4 Link Purpose
**Componente**: `Subscription.tsx:147`
**Problema**: `<a href="#">Não sabe qual escolher? Faça o teste em 30 segundos →</a>` — o `#` faz a página rolar para o topo, comportamento inesperado.
**Fix**: até o quiz existir, esconder o link ou converter em `<button disabled>` com mensagem clara, ou apontar para `/quiz` mesmo que retorne 404 (melhor que sair do contexto).

### HIGH-09 · Botão CartButton usa `bg-coffee-800` + `bg-gold-500` (paleta morta)
**WCAG**: 1.4.3 Contrast (e consistência visual)
**Componente**: `CartDrawer.tsx:50-65`
**Problema**: o badge dourado `bg-gold-500` (mapeado a `--olive #4A5237`) com `text-coffee-950` (mapeado a `--ink #1B1714`) sobre o badge: ratio **8.6:1** OK. Mas o contexto visual destoa do resto da Home (Header é `bg-ink/95`, o cart é `bg-coffee-800` = `bg-ink-soft`). Inconsistência visual gera dúvida — usuários cognitivamente impedidos podem não reconhecer como botão.
**Fix**: refactor completo do CartButton no sistema novo (ver CRIT-04).

## 5. Issues médios

### MED-01 · `prefers-reduced-motion` honrado globalmente, mas drawer ainda anima
**WCAG**: 2.3.3 Animation from Interactions (AAA, mas boa prática AA)
**Componente**: `CartDrawer.tsx:69-86`
**Problema**: `globals.css:510-519` aplica `transition-duration: 0.01ms` para tudo — bom. Mas o drawer usa `translate-x-full` via classe Tailwind, e a regra global cobre `transition-duration` mas o `transition-property` ainda anima opacity. Em prática reduz, mas validar.
**Fix**: testar com `prefers-reduced-motion: reduce` ativado e garantir que o drawer aparece instantaneamente.

### MED-02 · Header transparente sobre Hero pode falhar contraste durante o scroll
**WCAG**: 1.4.3 Contrast Minimum
**Componente**: `Header.tsx:39-43`
**Problema**: quando `headerOnLight === true` (Home, sem scroll), nav links usam `text-ink-soft` (#3A322C) sobre `bg-transparent` que mostra o `--bone` por baixo. Ratio 10.8:1 — passa. **Mas durante a transição** de bone para ink no scroll, há um momento em que o texto pode ficar ilegível. E em mobile, se o usuário começa a scrollar e a foto Hero passa por trás, o contraste varia.
**Fix**: adicionar leve `text-shadow` ou backdrop quando o header é transparente, ou fixar o background opaco mesmo no top.

### MED-03 · Footer copyright + termos com `text-bone-soft/60`
**WCAG**: 1.4.3 Contrast Minimum
**Componente**: `Footer.tsx:144,147`
**Problema**: `text-bone-soft/60` resolve para `rgba(235, 227, 213, 0.6)` sobre `--ink`. Ratio efetivo ~5.0:1 — passa marginal AA Normal, mas é apenas opacity sobre fundo escuro, leitores de tela ignoram opacity mas usuários com baixa visão sofrem.
**Fix**: usar `text-bone-soft` direto (sem /60) ou criar token `--ink-mute-on-dark` (#A39C92, ratio 5.7:1).

### MED-04 · Lang attribute correto, mas sem fallback regional para conteúdo italiano
**WCAG**: 3.1.2 Language of Parts
**Componente**: vários (`MDCCCXCVII`, `Dal 1897`, `Famiglia Zerbinatti`, `Treviso`)
**Problema**: `lang="pt-BR"` no `<html>` está correto. Trechos em italiano e latim (numerais romanos) podem ser pronunciados errado por TTS.
**Fix**: envolver com `<span lang="it">Famiglia Zerbinatti</span>`, `<span lang="la">MDCCCXCVII</span>`. Marginalia.

### MED-05 · `data` element em preço — bom semanticamente, sem currency code
**WCAG**: 1.3.1 Info and Relationships
**Componente**: `Cafes.tsx:177-183`
**Problema**: `<data value="89.90">R$ 89,90</data>` — `value` não inclui moeda. Parsers (e leitores) não sabem se é BRL ou USD.
**Fix**: usar microdata ou Schema.org Product/Offer no JSON-LD da página (verificar `lib/schema.ts`).

### MED-06 · Botões âncora (`<a href="#assinatura">`) com classe `btn` não têm `role="button"` mas comportamento de scroll-link
**WCAG**: 4.1.2 Name, Role, Value
**Componente**: `Hero.tsx:41-46`
**Problema**: visualmente botões, semanticamente links (que é correto — vão para uma âncora). Sem problema real, mas com `scroll-behavior: smooth` ativo, leitor de tela anuncia "link" mas o foco visual move-se em smooth scroll que pode confundir. Aceitável.
**Fix**: documentar o trade-off. Não bloqueador.

### MED-07 · Imagens lazy via `<Image>` sem dimensões explícitas em alguns casos
**WCAG**: 1.4.10 Reflow / boas práticas CLS
**Componente**: `Cafes.tsx`, `Story1897.tsx`
**Problema**: `fill` sem container com altura explícita causa CLS (layout shift). Embora não seja a11y diretamente, **CLS afeta usuários cognitivamente impedidos** que perdem o foco visual. O wrapper tem `aspect-[4/5]` — OK desktop, mas mobile usa `sm:aspect-[16/10] lg:aspect-[4/5]` (Cafes.tsx:133), os 3 breakpoints podem causar shifts.
**Fix**: validar CLS no Lighthouse, ajustar se >0.1.

## 9. Conformidade legal (Brasil)

- **LBI (Lei 13.146/2015) art. 63**: sítios da internet devem ser acessíveis. Padrão de fato: WCAG 2.1 AA via portaria do governo.
- **Decreto 9.296/2018**: regulamenta art. 63, exige conformidade para sítios públicos. Sítios privados não obrigados explicitamente, mas **risco de ação civil pública** existe (Defensoria Pública e MP têm precedentes).
- **Marco Civil da Internet + CDC**: textos legais (Cobrança recorrente, Cancele a qualquer momento) precisam ser **legíveis** — CRIT-02 expõe a marca a questionamento.

**Recomendação**: alcançar WCAG 2.1 AA pleno antes de lançamento comercial amplo. Adicionar **declaração de acessibilidade** no `/termos` ou `/acessibilidade` reconhecendo padrão atendido e canal de contato para reportar barreiras (boa prática Web Accessibility Initiative).
